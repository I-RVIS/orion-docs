Architecture
============

Orion is organized into four main modules, each handling a distinct part of the
robotic control pipeline.

---

Overview
--------

.. code-block:: text

                  ┌──────────────┐
                  │   Sensors    │  Camera, Microphone, ...
                  └──────┬───────┘
                         │
                         v
   ┌──────────┐   ┌──────────────┐   ┌───────────────────┐
   │  Target   │──>│  IK Solver   │──>│ Hardware Interface │──> Robot
   │ Position  │   │ + Collision   │   │  (Sim / Piper)    │
   └──────────┘   └──────┬───────┘   └───────────────────┘
                         │
                         v
                  ┌──────────────┐
                  │   Renderer   │  Real-time 3D view
                  └──────────────┘

---

Module: ``orion.motion``
------------------------

The motion module handles everything related to robot movement computation
and visualization.

**IKSolver**

The inverse kinematics solver converts end-effector targets (position + orientation)
into joint-space trajectories. It uses an iterative optimization loop powered by
``mink`` and the DAQP quadratic programming solver.

Key capabilities:

- Solve for a target SE3 pose with configurable error threshold
- Add collision avoidance constraints between geometry pairs
- Add joint velocity limits
- Add posture tasks (regularization toward a reference configuration)
- Returns a full trajectory ``[q0, q1, ..., qN]`` of joint configurations

.. code-block:: python

   from orion.motion import IKSolver
   from orion.types import AttachementFrameType

   solver = IKSolver(
       model_path="robot.xml",
       attachement_frame_name="eef_site",
       frame_type=AttachementFrameType.SITE,
       initial_configuration=[0.0] * 6,
       error_threshold=1e-3
   )

   # Add constraints
   pairs = IKSolver.generare_pairs_with_links(GEOM_NAMES, GEOM_NAMES)
   solver.add_collision_config(solver.create_collision_limit(pairs))
   solver.add_velocity_limit(solver.create_velocity_limit(velocity_limits))

   # Solve
   result = solver.solveIK(T_target=target_se3)
   # result = {"trajectory": [...], "steps": 42, "converged": True, "final_error": 0.0003}

**Renderer**

The renderer creates a real-time 3D visualization of the robot using MuJoCo.
It runs in a background thread and can publish frames to shared memory for
inter-process streaming (e.g. to a GUI or web viewer).

.. code-block:: python

   from orion.motion import Renderer

   renderer = Renderer(model_path="robot.xml", width=640, height=480)
   renderer.enable_shared_memory_stream("mujoco_stream")
   renderer.start_realtime_stream(fps=30)

   # Update the robot pose from the main thread
   renderer.data.qpos = [0.1, 0.2, 0.3, 0.0, 0.0, 0.0]

**CollisionChecker**

A standalone collision checker for validating joint configurations outside
the IK loop. Useful for custom planners or trajectory validation.

.. code-block:: python

   from orion.motion import CollsionChecker

   checker = CollsionChecker(
       model_path="robot.xml",
       joint_names=JOINT_NAMES,
       allowed_joints_list=["finger0", "finger1"],
   )

   collided, pairs = checker.check_collision(joint_position)

Module: ``orion.hardware``
--------------------------

The hardware module provides a pluggable abstraction layer for robot control.
You define behavior through callbacks — connect, disconnect, move, and read —
making it easy to swap between simulated and real hardware.

.. code-block:: text

   ┌──────────────────────────────────┐
   │    RobotHardwareInterface        │
   │                                  │
   │  set_connect_function(fn)        │
   │  set_disconnect_function(fn)     │
   │  set_move_function(fn)           │
   │  start_state_reader(fn, hz)      │
   │                                  │
   │  .state["joint"]                 │  ← updated by background thread
   │  .state["end_effector"]          │
   └──────────────┬───────────────────┘
                  │
        ┌─────────┴─────────┐
        v                   v
   SimHardwareInterface   PiperHardwareInterface
   (mock for testing)     (AgileX Piper via CAN)

**Predefined interfaces:**

- ``SimHardwareInterface`` — Instant mock updates, no hardware needed. Ideal for
  development and testing.
- ``PiperHardwareInterface`` — Controls the AgileX Piper robot arm via CAN bus
  using the ``piper_control`` library.

Both support context manager usage:

.. code-block:: python

   from orion.hardware.predefined_interfaces import SimHardwareInterface

   with SimHardwareInterface() as robot:
       robot.move(target_position)

Or manual connection:

.. code-block:: python

   robot = SimHardwareInterface()
   robot.connect()
   # ...
   robot.disconnect()

Module: ``orion.sensory``
-------------------------

The sensory module provides a generic interface for sensors. Like the hardware
module, it uses a callback-based architecture with background reading threads
and automatic reconnection support.

.. code-block:: text

   SensorHardwareInterface
        │
        ├── Camera           (video capture)
        ├── Microphone       (audio capture)
        └── MockSensor       (testing)

.. code-block:: python

   from orion.sensory.predefined_sensors import MockSensorInterface

   with MockSensorInterface() as sensor:
       while sensor.get_connection_status():
           # sensor reads data in background
           pass

Module: ``orion.types``
-----------------------

The types module provides strongly typed data structures for positions
and configurations.

**EndEffectorPosition**

Represents a 6-DOF end-effector pose (position + orientation in Euler angles).

.. code-block:: python

   from orion.types import EndEffectorPosition

   eef = EndEffectorPosition(
       position=[0.2, 0.0, 0.3],
       orientation=[0.0, 0.0, 0.0]   # Euler XYZ in radians
   )

   se3 = eef.to_mink_se3()       # Convert to mink SE3 for IK solver
   quat = eef.to_quaternion()     # Convert to [x, y, z, w]
   info = eef.beautify()          # Human-readable dict

**JointPosition**

Represents a joint-space configuration.

.. code-block:: python

   from orion.types import JointPosition

   joints = JointPosition(
       joint_length=6,
       joints_name=["joint1", "joint2", "joint3", "joint4", "joint5", "joint6"]
   )
   joints.position = [0.1, 0.2, 0.3, 0.0, 0.0, 0.0]

   qpos = joints.get_qpos()

**Enums**

- ``AttachementFrameType`` — Frame type in MuJoCo: ``SITE``, ``BODY``, or ``GEOM``
- ``KinematicType`` — Control mode: ``JOINT`` or ``EEF``
